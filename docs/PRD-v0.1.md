# ⚡ Xagent — 动力电池PACK创新智能体

## PRD v0.1 | 2026-06-11

---

## 1. 项目概述

### 1.1 定位

面向动力电池 PACK 结构设计与热管理领域的 **TRIZ 创新智能体**。为公司研发团队提供系统化的创新问题分析与方案生成能力，覆盖从问题定义到方案产出的完整 ARIZ 流程。

### 1.2 核心价值

| 痛点 | 解决方案 |
|------|----------|
| 工程师凭经验拍脑袋，方案缺乏系统性 | TRIZ ARIZ 流程引导，结构化分析 |
| 矛盾识别靠直觉，容易遗漏 | AI 辅助矛盾提取 + 矛盾矩阵自动匹配 |
| 跨领域经验难迁移 | 40 发明原理 × PACK 场景案例库 |
| 知识散落各处，复用困难 | 统一知识库，结构化存储 |
| 方案评审缺乏理论支撑 | TRIZ 原理追溯，增强方案说服力 |

### 1.3 用户

- **主要用户**：PACK 结构工程师、热管理工程师、创新项目组
- **使用场景**：研发办公环境，B/S 架构访问

## 3. 功能模块详细设计

### 3.1 模块① 系统组件分析

**目标**：分解问题系统，识别所有组件

**交互方式**：对话引导式

```
Agent: 请描述你要分析的系统是什么？
User:  方形铝壳电池 PACK 液冷系统

Agent: 系统的超系统（工作环境）是什么？
User:  整车底盘环境，温度范围 -30°C ~ 55°C

Agent: 系统的子系统有哪些？我先帮你列一些，你补充：
       [自动列出] 电芯模组、冷却板、冷却液、管路接头、BMS、箱体
       User: 再加上 导热垫、液冷板流道
```

**输出结构**：

```json
{
  "system_name": "方形铝壳电池PACK液冷系统",
  "supersystem": "整车底盘环境（-30°C~55°C）",
  "subsystems": [
    "电芯模组", "冷却板", "冷却液", "管路接头",
    "BMS", "箱体", "导热垫", "液冷板流道"
  ]
}
```

**LLM Prompt 策略**：
- System prompt 注入电池 PACK 领域知识（组件分类体系）
- 用 few-shot 示例引导用户逐步补充
- 自动识别遗漏的常见组件（基于知识库）

### 3.3 模块③ 功能建模

**目标**：建立系统功能模型图，标注功能类型

**功能类型定义**：

| 类型 | 符号 | 说明 |
|------|------|------|
| 有用功能 | ✅ | 正常的、期望的功能 |
| 不足功能 | ⚠️ | 功能存在但不够 |
| 过度功能 | 🔴 | 功能超出需要 |
| 有害功能 | ❌ | 不期望的副作用 |
| 缺失功能 | ➖ | 需要但不存在的功能 |

**交互方式**：Agent 基于前两步结果自动生成功能模型，用户审核

```
Agent: 基于前面的分析，我生成了功能模型：

       [电芯模组] ──✅ 产生热量──→ [环境]
       [电芯模组] ──✅ 提供电能──→ [高压输出端]
       [冷却液]   ──✅ 带走热量──→ [冷却板]
       [冷却板]   ──⚠️ 散热不足──→ [电芯模组]
       [导热垫]   ──⚠️ 导热系数偏低──→ [冷却板]
       [冷却液]   ──❌ 低温流动性差──→ [管路]

       请确认或补充/修正：
```

**输出结构**：

```json
{
  "functions": [
    {
      "source": "电芯模组",
      "target": "环境",
      "function": "产生热量",
      "type": "useful",
      "description": "电芯工作时产生焦耳热"
    },
    {
      "source": "冷却板",
      "target": "电芯模组",
      "function": "散热",
      "type": "insufficient",
      "description": "液冷散热能力不足以在高倍率工况下维持电芯温度均匀性"
    }
  ]
}
```

### 3.5 模块⑤ 功能建模问题总结

**目标**：从功能模型中提取所有问题点

**Agent 自动分析并分类**：

```
Agent: 基于功能模型，我识别出以下问题：

       🔴 过度功能（1项）：
          1. 电芯模组在快充时产生过多热量（>80°C 峰值）

       ⚠️ 不足功能（3项）：
          2. 冷却板散热能力不足（高倍率工况温差 >8°C）
          3. 导热垫导热系数偏低（现有材料 ≤6 W/m·K）
          4. 冷却液低温流动性差（-30°C 粘度急剧上升）

       ❌ 有害功能（2项）：
          5. 冷却液腐蚀铝制冷却板
          6. 蛇形流道造成末端电芯温度偏高

       请确认以上问题是否准确？有没有遗漏？
```

### 3.6 模块⑥ 因果链分析

**目标**：从表层问题追溯根因，建立因果链

**交互方式**：Agent 逐步追问，构建因果链

```
Agent: 让我们从"冷却板散热能力不足"开始分析因果链：

       表层问题：冷却板散热能力不足
       ├── 为什么？→ 流道内冷却液流速不够
       │   ├── 为什么？→ 水泵功率受限
       │   │   └── 为什么？→ 系统电压/功耗预算限制 ←【系统级约束】
       │   └── 为什么？→ 流道截面积过大
       │       └── 为什么？→ 蛇形流道设计导致沿程阻力大
       ├── 为什么？→ 冷却板与电芯接触热阻大
       │   ├── 为什么？→ 导热垫导热系数低
       │   └── 为什么？→ 接触面存在间隙/不平整
       └── 为什么？→ 冷却液比热容不足
           └── 为什么？→ 使用常规乙二醇水溶液

       关键切入点：
       ⭐ 流道设计优化（可避免系统级约束）
       ⭐ 导热界面材料升级（独立改进路径）
```

**输出结构**：

```json
{
  "causal_chains": [
    {
      "problem": "冷却板散热能力不足",
      "chain": [
        {"level": 1, "cause": "冷却液流速不够", "type": "direct"},
        {"level": 2, "cause": "水泵功率受限", "type": "system_constraint"},
        {"level": 3, "cause": "系统电压/功耗预算限制", "type": "root_cause"}
      ],
      "entry_points": [
        {"point": "流道设计优化", "feasibility": "high"},
        {"point": "导热界面材料升级", "feasibility": "medium"}
      ]
    }
  ]
}
```

### 3.7 模块⑦ 关键问题定义

**目标**：定义核心矛盾 + 理想最终状态（IFR）

**Agent 引导**：

```
Agent: 基于因果链分析，我定义了以下核心矛盾：

       ┌─────────────────────────────────────────────┐
       │ 技术矛盾（TC）                               │
       ├─────────────────────────────────────────────┤
       │ TC1: 提高散热效率 → 增加了系统重量和成本      │
       │ TC2: 减小流道尺寸 → 增加了流阻/降低流速       │
       │ TC3: 提高导热系数 → 材料成本增加              │
       └─────────────────────────────────────────────┘

       ┌─────────────────────────────────────────────┐
       │ 物理矛盾（PC）                               │
       ├─────────────────────────────────────────────┤
       │ PC1: 冷却液流量要大（散热好）又要小（能耗低）  │
       │ PC2: 导热垫要厚（绝缘好）又要薄（热阻小）     │
       └─────────────────────────────────────────────┘

       ┌─────────────────────────────────────────────┐
       │ 理想最终状态（IFR）                           │
       ├─────────────────────────────────────────────┤
       │ IFR1: 冷却系统自身不增加重量/体积/成本         │
       │       却能完美带走所有多余热量                  │
       │ IFR2: 电芯自己散热，不需要外部冷却系统          │
       └─────────────────────────────────────────────┘

       请确认矛盾定义是否准确？要优先解决哪个？
```

## 4. Skill 架构设计

### 4.1 Skill 清单

```
backend/skills/
├── __init__.py              # Skill 注册表
├── triz_ariz.py             # ARIZ 流程引擎（核心）
├── triz_matrix.py           # 矛盾矩阵 + 40发明原理
├── triz_evolution.py        # 8大技术进化法则
├── battery_pack.py          # PACK 领域知识（结构/热管理/BMS）
├── material_db.py           # 材料数据库（电芯/导热材料/冷却液）
├── patent_search.py         # 专利检索分析
└── wiki_knowledge.py        # 知识库管理（保留原 llm_wiki）
```

### 4.2 Skill 间协作

```
用户输入问题
    ↓
[triz_ariz] 识别系统 → 调用 [battery_pack] 获取领域知识
    ↓
组件分析 + 接触分析 → 调用 [material_db] 获取材料参数
    ↓
功能建模 + 问题总结
    ↓
因果链分析 → 调用 [battery_pack] 验证因果关系合理性
    ↓
矛盾定义 → 调用 [triz_matrix] 匹配发明原理
    ↓
方案生成 → 调用 [patent_search] 检索相关专利佐证
    ↓
输出方案卡片
```

## 5. 数据模型

```
projects/
├── id: uuid

├── contacts: [ ... ]       # 接触关系
├── functions: [ ... ]      # 功能模型
├── structure: { ... }      # 结构参数
```

## 7. 技术实现要点

### 7.1 后端

- ARIZ 流程引擎：**状态机**，维护当前步骤 + 各步骤数据
- 矛盾矩阵：预加载到内存，支持快速查询
- LLM 调用：system prompt 注入领域知识，few-shot 示例引导
- 项目持久化：SQLite（现有架构扩展）

### 7.2 前端

- 流程状态管理：Zustand 或 React Context
- 可视化：D3.js / React Flow（功能模型图、因果链）
- 矛盾矩阵：Canvas 或 SVG 绘制

### 7.3 LLM 策略

```
System Prompt 结构：
├── 角色定义（TRIZ 专家 + 电池 PACK 专家）
├── ARIZ 流程规则（当前步骤 + 输入要求 + 输出格式）
├── 领域知识注入（从知识库动态检索）
├── 输出格式约束（JSON schema）
└── few-shot 示例（每步 2-3 个）
```
